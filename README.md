# Validity_Fairness_Testing

## Publication

This repository is associated with the following paper:

**_"Is Validity a Meaningful Metric in Fairness Testing?"_**

(*Under double-blind review*)

---

## About this repository

This repository provides the implementation, data, and results used to study the role of validity in individual fairness testing (IFT).


The repository is structured as follows:

- `datasets_original/`  
  Original datasets used in the experiments, along with a configuration file for dataset settings.

- `datasets_prepared/`  
  Preprocessed datasets used for evaluation (accuracy, IFr, valid-IFr),  
  as well as occurrence tables derived from training data for validity checking.

- `IDIs/`  
  Individual discriminatory instances detected by fairness testing (AFT), including:
  - raw IDIs,
  - valid IDIs,
  - invalid IDIs.

- `IFT/`  
  Implementation of AFT used to detect IDIs.

- `models_trained/`  
  Models trained on the training datasets.

- `models_retrained/`  
  Models retrained using detected IDIs.

- `results/`  
  Experimental results generated by the REDI framework,  
  along with scripts for analysis (e.g., partial correlation computation).

- `results_paper/`  
  Results reported in the paper. 
  Specifically, it includes the outputs produced by running:
  ```
  python REDI.py --model all --dataset all --protected all --runs 10
  ```
  along with the corresponding analysis results.

- `REDI.py`   
  Main script of the experimental pipeline based on the REDI framework described in the paper.
  It orchestrates the full workflow from data preparation to retraining and evaluation.


The directories `datasets_prepared/`, `IDIs/`, `models_trained/`, `models_retrained/`, and `results/`
contain the necessary scripts and folder structures, but do not include the experimental data by default.

Running `REDI.py` generates the required data and stores the results in these directories.

The results reported in the paper are provided separately in `results_paper/`.

Each directory contains a dedicated `README.md` with further details.

## Usage

The main experimental pipeline can be executed using:

```
python REDI.py --model MODEL --dataset DATASET --protected ATTRIBUTE --runs N
```


---

### Parameters

- `--model`: Model name (`SVM`, `MLP`, `RanForest`, or `all`)
- `--dataset`: Dataset name (`Adult`, `Bank`, `Credit`, or `all`)
- `--protected`: Protected attribute (`age`, `sex`, `race`, or `all`)
- `--runs`: Number of repeated executions (default: 1)

---

### Supported Scenarios

Not all combinations of parameters are supported.
The experiments are limited to predefined scenarios used in the paper.

The supported scenarios are:

- (SVM, Adult, age)
- (SVM, Credit, age)
- (SVM, Credit, sex)
- (MLP, Adult, age)
- (MLP, Adult, race)
- (MLP, Adult, sex)
- (MLP, Bank, age)
- (MLP, Credit, age)
- (MLP, Credit, sex)
- (RanForest, Bank, age)
- (RanForest, Credit, age)
- (RanForest, Credit, sex)

Specifying unsupported combinations will result in an error.

---

### Using `all`

Each argument (`--model`, `--dataset`, `--protected`) accepts the value `all`.

- If `all` is specified, all matching predefined scenarios are executed.
- The script does not generate new combinations beyond the supported scenarios.

---

### Examples

Run the scenario (SVM, Adult, age) once:

```
python REDI.py --model SVM --dataset Adult --protected age --runs 1
```

Run all scenarios for the Credit dataset once each:

```
python REDI.py --model all --dataset Credit --protected all --runs 1
```

Run all predefined scenarios 10 times each:

```
python REDI.py --model all --dataset all --protected all --runs 10
```

## Pipeline Description

Executing `REDI.py` runs the experimental pipeline used in this study.
While the pipeline follows the methodology described in the paper,
it also includes several implementation-level steps required for execution.

For each selected scenario (model, dataset, protected attribute), the following steps are performed:

1. Split the dataset into training and test sets  
2. Construct occurrence tables for validity checking based on the training dataset  
3. Generate test inputs for IFr  
4. Generate test inputs for valid-IFr  
5. Train the original model  
6. Perform fairness testing (AFT) to detect IDIs and classify them into valid and invalid sets  
7. Retrain the model using the detected IDIs with varying levels of validity  

The results are stored in the `results/` directory after execution.
